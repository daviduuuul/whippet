# deps-audit value eval — precision/recall of the native-equivalent check

**Question.** The H1 live experiment found that on trivial native-equivalent tasks a modern
model doesn't reach for a needless dependency anyway. So whippet's value isn't babysitting a
fresh choice — it's the deterministic engine catching needless deps in a *real, already-written*
`package.json`. This eval measures exactly that: does `deps-audit` correctly flag deps that a
Node built-in replaces, **without false-positives**, across realistic dependency choices?

## Method

- **Corpus** (`corpus.json`): 106 cases, 780 labeled deps. Generated by 10 independent agents,
  one per project stack (Express API, React SPA, Next.js, CLI, bot, Electron, build tooling, ETL,
  testing, legacy), each told to list a real project's dependencies and label every one
  `needless_native` (a Node built-in fully replaces it at this `engines.node` floor — with the API
  and the Node major it landed) or `legit` (multi-purpose / no native / native only above the
  floor). Floors are spread across `>=16/18/20/22` so the engine's version-gating is exercised.
- **Scoring** (`eval.js`): materialize each case's `package.json` + a source file that imports
  every dep (so the *unused* check stays silent and we isolate the **native** check), run
  `deps-audit`, compare its `native` findings to the labels. `node eval.js corpus.json`.
- **Verification**: every candidate swap surfaced as a recall gap was adversarially fact-checked
  against Node release history (stable-vs-experimental major) before being added — see
  `whippet/scripts/deps-audit.js`. A wrong swap is a false-positive, which is worse than a miss.

## Result

| | precision | recall | FP | FN |
|---|---|---|---|---|
| **before** | 92.0% | 81.7% | 19 | 49 |
| **after**  | **96.6%** | **85.8%** | **8** | 38 |

**What changed (verified swaps only):**
- **+7 native swaps** (single-purpose, native stable at a clean major boundary):
  `lodash.clonedeep` / `fast-copy` → `structuredClone` (17), `isarray` → `Array.isArray`,
  `es6-promise` → `Promise`, `p-defer` → `Promise.withResolvers` (22),
  `minimist` → `util.parseArgs` (20), `querystring` → `node:querystring` / `URLSearchParams`.
  These added recall with **zero** new false-positives on the corpus.
- **Precision fix:** `node-fetch` / `cross-fetch` floor `18 → 21`. Global `fetch` is *experimental*
  on Node 18–20 and stable only from 21, so flagging them below 21 was wrong — this removed 11 FPs.
- **Deliberately rejected** (false-positive risk the major-granularity gate can't avoid):
  `ncp`/`fs.cp` (stable only 22.3), `glob`/`fs.globSync` (stable 24, feature-rich),
  `chalk`/`util.styleText` (mid-major 22.13, lacks rgb/hex/template), `deep-equal` (its default
  loose `==` mode differs from `util.isDeepStrictEqual`).

## Honest reading of the remaining 8 FPs

They are not naked false-positives — they are deps the engine flags **with a caveat note**, on a
usage it cannot see from the declaration:
- `uuid` (4): flagged with *"native covers v4 only"*; the corpus cases happen to use v5/`validate`.
  The declaration can't reveal the version, so the engine flags + warns rather than stay silent.
- `dotenv` on Node 22 (4): `--env-file` covers the common case but not `${VAR}` expansion; the
  finding now carries that caveat. Whether it's needless depends on whether expansion is used.

So the *harmful* false-positive rate (a confident wrong flag) is effectively 0 on this corpus; the
8 are advisory findings with the right caveat. Corpus labels are agent-generated (debatable on the
margins); the **swaps** are the adversarially-verified part. Reproduce: `node eval.js corpus.json`
(writes `disagreements.json`), or `node eval.js --selftest`.
