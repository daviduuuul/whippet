# A/B run — 2026-06-19, Claude Opus 4.8

First real paired A/B on the fixture harness. Replaces the v1 self-reported seed
(`opus-4.8-2026-06.json`, removed) with objectively scored runs. Raw observations:
`runs.jsonl` (75 lines).

## Method

**Model-instruction A/B** (not a full Claude-Code-session A/B): one model
(`claude-opus-4-8`), three arms that differ **only** by the discipline prepended
to an otherwise identical prompt:

- **off** — solve it as a competent dev normally would; explicitly ignore any lean/whippet discipline.
- **baseline** — the one-line nudge from METHODOLOGY ("minimal code that works; reuse existing/stdlib/native before adding code or deps; report tersely").
- **whippet** — the full ladder + never-cut-validation/security + terse reporting.

5 fixtures × 3 arms × **5 runs** = 75 candidates. Candidates were generated by
subagents (blind to the hidden graders), then scored **deterministically** by
`scripts/bench-score.js` — the numbers do not depend on anything the generating
agent reported. Pre-registered metrics: correctness (hard gate), `loc_added`,
`files_added`, `deps_added`, `reused`, reported per category.

## Findings (honest)

1. **Correctness parity — 25/25 for every arm, every category**, including
   `trap_overcut` (validation): even the *off* arm kept the input guard. On a
   strong 2026 model, correctness on these tasks is universal; whippet does not
   improve it because the model is already correct.
2. **Diff size is the only discriminator.** Mean LOC added: whippet **3.9**,
   baseline **3.6**, off **19.2** — the disciplined arms produce ~5× smaller
   diffs than an un-nudged agent, at equal correctness. Per category, *off* is
   3–10× larger everywhere (e.g. `trap_yagni` 0.4/1.8 vs 17.6; `trap_overcut`
   14/10.6 vs 47.2 — validation legitimately needs code, so the lean arms keep
   it *and* stay smaller).
3. **whippet ≈ baseline.** The full plugin does **not** beat the one-line nudge
   on these fixtures (3.9 vs 3.6 LOC; both 25/25 correct; deps 0/0). Per
   METHODOLOGY's own rule, that null is published, not hidden: on small,
   single-function tasks the elaboration buys nothing the one-liner doesn't.
4. **No arm added a dependency** (0/0/0). Reuse of the flagged helper (`trap_reuse`
   only): **5/5 every arm** — a tie. An earlier cut of this run reported reuse as
   whippet 15/15 vs baseline 13/15, whippet's one apparent edge; that was a metric
   artifact. The `reused` boolean had been recorded across three categories whose
   right answer is three *different* rungs — reuse existing code (`trap_reuse`),
   take the stdlib (`trap_stdlib`), don't build it (`trap_yagni`) — so the 15
   denominator pooled rungs that aren't reuse, and the whole 15-vs-13 gap was 2
   runs in `trap_stdlib`. Scoped to where reuse means reuse, the edge disappears:
   no metric separates whippet from baseline.

## Caveats (what this does NOT show)

- Small **single-function** fixtures, **n=5** per cell → wide Wilson CIs; the
  correctness ceiling (25/25) means LOC is the only axis with spread.
- **Model-instruction** A/B, not full Claude-Code sessions (no tool-use, no
  multi-turn, no session hook). It isolates the *discipline-as-prompt* effect.
- One model, one day. Temperature-0 is not deterministic; re-runs will vary.
- The reviewable-diff gap (whippet/baseline ≪ off) is consistent with DORA p.58's
  small-batch lever but measured on toy tasks, not delivery instability.

A bigger, multi-step fixture set and a full-session arm are the next step before
any headline claim rests on this.

## Verdict — what we can and can't claim (added after adversarial review)

Read straight, this run argues *against* the product as a prompt:

- **The content doesn't beat a one-liner.** whippet 3.9 vs baseline 3.6 LOC,
  25/25 each, 0 deps each. On the artifact, the elaborate SKILL ties a one-line
  "write less code." It does not win.
- **The flagship safety claim doesn't win either.** On `trap_overcut` — the
  category that tests "never cut validation" — whippet wrote *more* (14 LOC) than
  baseline (10.6), and baseline kept the guard too (both 5/5). We don't beat the
  one-liner on the thing we brag about most.
- **The "~5×" is off-vs-discipline, and the baseline gets the same 5×.** It's the
  value of *having any discipline*, not of whippet. Presenting it as whippet's win
  is the manifest-self-report sin transposed onto attribution. (README fixed.)
- **The stats over-sell.** Wilson/McNemar/Wilcoxon on n=5, one model, one day,
  single-turn toy fixtures: the CIs only separate off-from-rest, the one
  comparison that needed no study. Rigorous-looking, can't conclude what the
  product needs it to.

**So where's the value, honestly?** Not a measured artifact edge. The only thing
that could justify whippet over pasting the discipline into a repo's `CLAUDE.md`
is *persistence across compaction* — and that doesn't hold either: **`CLAUDE.md`
is re-injected after compaction by design** (it's memory, not part of the
summarized transcript), so a pasted discipline survives compaction for free.
whippet's `SessionStart(compact)` re-anchor does the same thing, no better.

What whippet genuinely offers over a `CLAUDE.md` paste is **packaging, not a
mechanism advantage**: install once → applies in every repo (CLAUDE.md is
per-project), mode control, the review/simplify/ledger commands, and a curated,
guard-tested discipline. Real and modest. *Not* "a measured cut in code or drift."

## The drift test (the experiment that would settle it)

Can't be faked at n=5 or run headless here — it needs a real long session with
real compaction. Protocol, for whoever runs it:

1. Three arms, same model, same task stream. **A:** discipline pasted once into
   the first turn, never repeated. **B:** in `CLAUDE.md`. **C:** whippet installed
   (re-anchors on compact). A-vs-rest isolates *persisting at all*; **B-vs-C
   isolates packaging from any real mechanism** — that's the comparison that matters.
2. Drive ~40–60 turns of mixed work until at least **two auto-compactions** fire.
3. After each compaction, inject a fixture trap (reuse / stdlib / over-cut) and
   score the output with `bench-score.js`.
4. Metric: post-compaction lean-compliance rate per arm, ≥5 sessions, with CIs.

Prediction from how the harness works: **C ≈ B ≫ A** — whippet beats *forgetting
to persist*, ties *persisting via CLAUDE.md*. If that holds, the honest pitch is
what the README now says: convenience and control, not a drift win. If C beats B,
that's a real finding worth publishing. Until measured, no persistence claim ships.

### Drift pilot — run 1 (manual, inconclusive)

Ran session 1 of the protocol by hand on Opus 4.8: three isolated arms (A paste-once
/ B CLAUDE.md / C whippet), two `/compact`s each, a trap scored after each
(`stdlib`, then `reuse`). Raw: `benchmarks/drift/drift-runs.jsonl` (6 obs).

Result: **A = B = C, 2/2 lean-compliant each.** The traps sat at the ceiling — even
arm A, the paste-once that is *not* re-injected after `/compact`, stayed lean. So run 1
doesn't separate the vehicles; it's a null, and an uninformative one (the
discriminating traps — `yagni`, `overcut` — never got a clean run).

Why it stopped here, stated plainly: the full protocol (≥5 sessions × 3 arms with
real compaction) is heavy by hand and **not** automatable headless without losing the
thing it measures — a fresh `claude -p` session has no compaction and collapses back
into the A/B we already ran; the auto-mode classifier (correctly) blocks spawning
autonomous headless agents. So the pilot is what we have.

It changes nothing about the verdict: the A/B tie, CLAUDE.md's free re-injection after
compaction, and this ceiling pilot all point the same way — **packaging, not a
mechanism edge.** The C ≈ B prediction stands un-refuted; no claim ships beyond the
pilot. Anyone wanting the real number runs the protocol by hand over a long session.
