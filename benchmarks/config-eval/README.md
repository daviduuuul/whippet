# config-audit value eval — detection vs false-positives on realistic configs

**Question.** config-audit's whole pitch is "catch the config drift the schema can't, *without
false alarms*." Does it? This eval measures detection (recall) on planted drift and, just as
important, the false-positive rate on valid-but-tricky config.

## Method

- **Corpus** (`corpus.json`): 82 scenarios generated by 9 independent agents (one per drift
  theme + clean-config themes). Each is a `settings.json` (± `settings.local.json` / `.mcp.json`)
  with a ground-truth list of planted drift (kind + the exact offending token) — or an empty list
  for a fully-valid config. 32 scenarios are entirely clean (pure false-positive test).
- **Scoring** (`eval.js`, has `--selftest`): materialize the files, run config-audit, match
  findings to labels. Only the **content** checks are scored — the filesystem-dependent ones
  (hook/statusLine *script missing*, plugins, manifests, marketplaces, components, backups) need
  an install tree the corpus doesn't model, so their findings are out of scope here.
- **Verification**: the disagreements were checked against **authoritative Claude Code docs**
  (`code.claude.com/docs/en/hooks`, `/settings`) — this is what exposed the false-positive bugs.

## What the eval exposed — and the fixes

The eval surfaced **4 real false-positive bugs** (valid config wrongly flagged), all from closed
sets that had drifted behind Claude Code:

| bug | cause | fix |
|---|---|---|
| valid hook events flagged | `HOOK_EVENTS` listed 9 of the 30 documented events → `PostToolUseFailure`, `PermissionRequest`, … flagged as "unknown" | synced to the full 30; **hybrid** check: a near-miss of a known event is an `error` (typo), an unknown-but-far name is only a `warning` (could be a newer event), so the autonomous advisory never hard-fails on a future event |
| `type: mcp_tool` flagged | `HOOK_TYPES` missing `mcp_tool` | added it |
| matcher `"*"` flagged | the matcher was always `new RegExp`-validated, but `*`/`""`/exact-pipe forms are not regexes | only true-regex matchers are compiled now |
| `timeout: 2.5` flagged | required a positive **integer** | timeout is any positive **number** (docs) |

Plus **2 recall improvements**: a typo sitting next to the correct key is now flagged as a dead
near-duplicate (was suppressed); and the typo distance is Damerau (a transposition like
`modle`→`model` is one edit).

## Result

| | recall | real false positives |
|---|---|---|
| before fixes | 94.2% | **5** (the bugs above, on valid config) |
| **after fixes** | **96.5%** (83/86) | **0** |

The 4 findings the eval still counts as "FP" are **corpus mislabels, not engine errors**:
all are `autoUpdatesChannel: "rc"`, which the docs confirm is *invalid* (only `stable`/`latest`) —
config-audit is right to flag it. The 3 remaining misses are: one corpus mislabel (`timeout: 2.5`
is valid, so not flagging it is correct), and two documented limits — a wrong key *name* that is
not a near-miss of any real key (`enviroment` for `env`), and a deny glob that *subsumes* an allow
glob without being an exact string match (needs glob-containment, out of scope for the exact-match
overlap check).

**Dogfood:** run against a real local hub config (`$CLAUDE_CONFIG_DIR`) → 0 errors, 0 warnings,
0 hook/key/enum findings. No false alarms in the wild.

Reproduce: `node eval.js corpus.json` (writes `disagreements.json`), or `node eval.js --selftest`.
